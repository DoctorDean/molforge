## Code of Conduct

This project follows the [Contributor Covenant](CODE_OF_CONDUCT.md). By participating, you agree to uphold it.

## Ways to contribute

- **Report bugs** via the [bug report template](.github/ISSUE_TEMPLATE/bug_report.yml).
- **Propose features** via the [feature request template](.github/ISSUE_TEMPLATE/feature_request.yml).
- **Improve docs** — typo fixes are welcome, no issue needed.
- **Add a wrapper** for an external engine (docking, folding, MD, scoring).
- **Add a parser** for an additional file format.
- **Add a plugin** that demonstrates the registry API.

## Development setup

Requires Python 3.10+.

```bash
git clone https://github.com/OWNER/biocore.git
cd biocore

python -m venv .venv
source .venv/bin/activate     # Windows: .venv\Scripts\activate

pip install -e ".[dev,docs]"
pre-commit install
```

Run the full local check before pushing:

```bash
ruff check .
ruff format --check .
mypy src
pytest
```

## Workflow

1. **Open an issue first** for non-trivial changes so we can align on scope.
2. **Fork** the repo and create a feature branch from `main`:
   `git checkout -b feat/<short-description>` or `fix/<short-description>`.
3. Make focused commits — see [commit conventions](#commit-and-pr-conventions).
4. Update `CHANGELOG.md` under `[Unreleased]` with a brief entry.
5. Push and open a pull request against `main`.
6. Ensure CI is green and address review feedback.

## Coding standards

- **Style:** `ruff` for linting + formatting. Run `ruff format .` and `ruff check --fix .`.
- **Types:** every public function and class must have full type hints. We run `mypy --strict` in CI.
- **Docstrings:** Google style. Public APIs require a one-line summary, `Args`, `Returns`, and `Raises` sections where applicable.
- **Imports:** sorted by `ruff` (isort rules). No `from x import *`.
- **Naming:** `snake_case` for functions/variables, `PascalCase` for classes, `UPPER_SNAKE_CASE` for constants.
- **Public API:** anything not prefixed with `_` is public and subject to semver.

## Testing

- Tests live in `tests/`, mirroring the `src/biocore/` layout.
- Use `pytest`. Property tests with `hypothesis` are encouraged.
- Mark slow tests with `@pytest.mark.slow` and integration tests with `@pytest.mark.integration`.
- Aim for >90% line coverage on new code.

```bash
pytest                         # full suite
pytest -m "not slow"           # quick
pytest --cov=biocore           # with coverage
pytest tests/unit/core/        # one subpackage
```

## Documentation

- API docs are generated by `mkdocstrings` from docstrings.
- Tutorial notebooks live in `notebooks/walkthroughs/`. Keep them runnable end-to-end.
- Build docs locally with: `mkdocs serve`.

## Commit and PR conventions

We use **Conventional Commits**:

```
<type>(<scope>): <short summary>
```

Types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`.

Examples:
- `feat(core): add Atom.element property`
- `fix(io): handle insertion codes in PDB parser`
- `docs(readme): clarify install extras`

PRs should:
- Have a Conventional Commit-style title.
- Reference the issue they close (`Closes #123`).
- Include tests for new behavior.
- Update `CHANGELOG.md` under `[Unreleased]`.

## Releasing (maintainers only)

1. Bump version in `src/biocore/__init__.py`.
2. Move `[Unreleased]` entries to a new versioned section in `CHANGELOG.md`.
3. Commit: `chore(release): vX.Y.Z`.
4. Tag: `git tag -a vX.Y.Z -m "Release X.Y.Z"` and push tags.
5. The `release.yml` workflow publishes to PyPI on tag push.
